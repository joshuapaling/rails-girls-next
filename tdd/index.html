TDD Exercise

This exercise was run for me by a great guy named Steve Hardisty when we both worked at REA Pty Ltd. It is intended as an exercise to teach you what we're talking about when we say Test Driven Development. It also involves pair (or in our case group) programming.

Overview

You will be writing a Roman Numeral converter using test driven development (TDD). The converter will take a Roman Numeral (i.e. "I", "IV", or "XX") and convert it into a number (i.e. 1, 4, 200).

  • How Roman numerals work: http://www.onlineconversion.com/roman_numerals_advanced.htm
  • Split into groups of 3, each with a computer.
  • Each group will have a hot seat with laptop, keyboard and Sublime Text.
  • A mentor will provide the initial code (below).
  • Within each group, everybody rotates chairs so the next person is at the keyboard, each time:
    • A new test is written, or
    • A test goes from failing to passing.

0. Initial Code

The mentor types this out:

def roman(n)
  return "?"
end

require "minitest/spec"
require "minitest/autorun"
require "minitest/pride"

describe "roman" do
  it "converts the number 1 to the string I" do
    roman(1).must_equal "I"
  end
end

Run the test using Ctrl-B in Sublime Text or typing the following at the prompt:

ruby roman.rb

Outputs:

roman#test_0001_converts the number 1 to the string I [tdd1.rb:11]:
Expected: "I"
  Actual: "?"

1 tests, 1 assertions, 1 failures, 0 errors, 0 skips

Your code is now red - one or more of the tests fail.

1. Write code

Everybody changes seats (it's like musical chairs!). A new person (not the mentor) is now at the keyboard, they try to make this test pass. They may do this however they see fit, with feedback and hints from the other group members. It's fine if the change is just an extra if statement. In fact, that's encouraged, to put the pressure on the person writing the test to try harder to break it.

Your code is now green all of your tests pass. Commit your code at this point:

$ git commit -m "your commit message here"

And push your code up to github:

$ git push origin master

2. Write test

Once you have your tests passing (again, it doesn't matter how "good" the code is), everybody changes seats again. The new "driver" (person writing the code), should write another test. You can write any test you like, so long as it moves you towards the goal of converting Roman Numerals. In general your tests will have a logical progression, but you'll find you might need to go back and add tests, that's completely fine! Again, the other group members ("ovservers") should give suggestions and hints (but they should NOT take over the keyboard, they're observing!)

Your code is now red again.

3. Rinse and Repeat!

Now you have the idea, you can keep repeating steps 1 and 2, making sure to continue switching the driver each time.

Whenever you feel like some of the code needs to be refactored (cleaned up and made easier to read), stop and work through the refactor. You should only refactor when your code is green, you should work through the refactor as a group, and your code should be green by the end of the refactor. Make sure you commit after a refactor.
